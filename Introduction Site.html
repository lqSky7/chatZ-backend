⚡ WebSocket Protocol ▾

The WebSocket URL is derived from VITE_API_URL by stripping /api and replacing the scheme: http:// → ws://, https:// → wss://.

Client → Server actions

actionPayloadEffect
identify{ userId }Links this WS connection to a user ID for targeted push
join{ roomId }Subscribes to a room; server immediately sends full history

Server → Client events

actionPayloadWhen
history{ messages[] }Immediately after join
newMessage{ message }When any member sends a message to the room
identified{ userId }Confirmation after identify
joinRequestSubmitted{ request }Pushed to room creator when someone requests to join
joinRequestApproved{ room }Pushed to requester when creator approves
joinRequestRejected{ room }Pushed to requester when creator rejects
Note: Room messages are broadcast to all WS clients subscribed via the rooms Map. Targeted notifications use the userSockets Map — this is why identify must be called at login.
🔨 How It Was Built ▾
1

Requirements & Protocol Design

Identified need for group rooms with approval flows + private DMs. Defined two-channel architecture: REST for persistence, WebSocket for real-time delivery.

2

Database Schema

Created five normalized tables. Used ON CONFLICT DO NOTHING for idempotent inserts. chatrooms.type discriminator keeps groups and DMs in one table.

3

Backend: Express + WebSocket

Thin Express layer for REST. ws library WebSocket server shares the same HTTP server. Two in-memory Maps route broadcast vs targeted push events.

4

Auth & Room Join Flow

bcrypt hashes passwords. Group room join is a state machine: create-if-new → auto-approve creator → send pending request → creator approves/rejects → WS pushes result.

5

Frontend Service Layer

api.js wraps every REST call with logging and error normalization. websocket.js implements a listener-registry pattern with automatic reconnect and pending join queue.

6

Broadcast DM Utility

sendBroadcast() in api.js iterates over recipient IDs, finds or creates a DM room with each, then sends the message — partial failures tracked without stopping the loop.

🚀 Getting Started ▾
1

Clone / unzip the project

Place the project folder somewhere on your machine. It contains backend/ (Node.js server) and src/ (Vite frontend).

2

Set up PostgreSQL

Install PostgreSQL and create a database. Copy .env.example to .env and fill in DATABASE_URL.

# Example DATABASE_URL DATABASE_URL=postgresql://myuser:mypassword@localhost:5432/chatz
3

Install backend dependencies

cd backend npm install
4

Start the backend server

npm start # Server running on http://localhost:3000 # Database initialized successfully

The server auto-creates all tables on first run. Leave this terminal open.

5

Install and start the frontend

npm install npm run dev # ➜ Local: http://localhost:5173
6

Open the app

Visit http://localhost:5173 in your browser. You'll see the login/register screen.

💬 Using the Chat Features ▾

Register and log in

On first visit, click Register and create a username and password. Your session is stored in localStorage (userId + name).

Join or create a group room

Enter any room code and a password. If the code doesn't exist yet, you become the room creator and are joined immediately. If it exists:

  • Enter the correct room password
  • A join request is sent to the creator
  • The creator sees a real-time notification and can approve or reject
  • You're notified instantly via WebSocket when reviewed

Start a Direct Message (DM)

Use the DM section in the sidebar. Enter a target user's ID or username to open a private channel. ChatZ finds or creates the DM room automatically.

Sending messages

Type in the message input and press Enter. Messages are persisted to PostgreSQL and broadcast in real time. When you rejoin a room, full history is replayed.

Managing room members (creators only)

As a room creator you can view the member list and remove users via the room settings panel. Removal calls DELETE FROM room_members — the user loses access immediately.

🛠️ Troubleshooting ▾
❌ "password authentication failed for user"
Cause: Wrong credentials in DATABASE_URL
Check your backend/.env — username, password, host, and database name must all match your PostgreSQL setup.
❌ "Connection refused" on port 5432
Cause: PostgreSQL is not running
Start PostgreSQL: sudo service postgresql start (Linux) or open the Postgres app (Mac).
❌ WebSocket connects then immediately disconnects
Cause: VITE_API_URL points to wrong host or port
Make sure VITE_API_URL=http://localhost:3000/api in your root .env and the backend is running on port 3000.
❌ "User already exists" on register
Cause: The username is taken (UNIQUE constraint on users.name)
Try a different username, or log in instead if the account is yours.
❌ Join request never arrives at creator
Cause: Creator's browser has not sent the identify WS action
The creator must be logged in and have the app open. The identify action is sent on WS connect — ensure the creator refreshed after logging in.
❌ Messages not appearing in real time
Cause: WS not subscribed to the room or connection dropped
Reload the page to reconnect. Check the browser console for [WS] Disconnected logs. If behind a proxy, ensure it supports WebSocket upgrades.
❌ Frontend shows blank page / Vite error
Cause: Missing npm install or node_modules
Run npm install in the project root (not inside backend/), then npm run dev.
❌ "EADDRINUSE" error when starting backend
Cause: Port 3000 already in use
Kill the old process: lsof -ti:3000 | xargs kill, or set PORT=3001 in backend/.env and update VITE_API_URL accordingly.
